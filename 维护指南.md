# KsText 维护指南

这份文档是给仓库维护者用的。

目标只有一个：
把 `packs/` 里的文本包和根目录的 `index.json` 维护正确，让 mod 可以正常搜索、选择、下载、加载。

如果你不想研究原理，只想把仓库维护对，照着本文一步一步做就行。

## 1. 仓库结构

仓库当前只认这两个核心位置：

```text
KsText/
├─ index.json
└─ packs/
   ├─ 文案包A.json
   ├─ 文案包B.json
   └─ 文案包C.json
```

含义：

- `packs/`
  - 这里放每一个 killsay 文本包
  - 一个文件就是一个文本包
- `index.json`
  - 这里放所有文本包的目录
  - mod 搜索列表时先读它
  - 玩家点某个包时，再按里面的 `downloadUrl` 去下载对应的 `packs/*.json`

所以你必须记住一句话：

**新增、删除、改名、修改文本包后，`index.json` 也要同步改。**

## 2. 最重要的规则

### 文件名规则

- 每个 killsay 文本包都用一个单独的 `.json` 文件
- 文件名直接用中文
- 文件名尽量和这个文本包的名字一致
- 不要随便混用全角半角空格
- 不要今天叫 `情话版.json`，明天又改成 `情话版1.json`，除非你确定要改

### 作者规则

- 文本里写了作者，就填真实作者名
- 文本里没写作者，就统一填 `佚名`

### 分支规则

这个仓库现在远端主分支是 `master`。

所以：

- `index.json` 要从 `master` 读取
- `downloadUrl` 也最好统一写成 `master`

不要一部分写 `master`，一部分写 `main`。

以前已经出过这个错：

- `index.json` 在 `master`
- 但是 `downloadUrl` 写成了 `main`
- 结果列表能同步，点进去下载包却 404

所以以后维护时，**分支必须统一**。

## 3. 一个文本包文件长什么样

每个 `packs/*.json` 文件结构都应该像这样：

```json
{
  "id": "情话版",
  "name": "情话版",
  "entries": [
    "{name} 你像晚风一样温柔",
    "{name} 今天先陪你到这里",
    "{name} 你的结局被我收下了"
  ]
}
```

字段说明：

- `id`
  - 这个包的唯一标识
  - 最好和文件名、名字保持一致
- `name`
  - 给玩家看到的名字
- `entries`
  - 真正的 killsay 文本列表
  - 一行一个句子

注意：

- `entries` 里每一条都必须是字符串
- 不要在最后多写奇怪的注释
- JSON 不能带尾逗号
- 文本里如果要引用玩家名，继续用 `{name}`

## 4. `index.json` 长什么样

`index.json` 结构大概像这样：

```json
{
  "schemaVersion": 1,
  "generatedAt": "2026-05-26T00:00:00Z",
  "packs": [
    {
      "id": "情话版",
      "name": "情话版",
      "author": "佚名",
      "summary": "温和一点的文案包",
      "language": "zh-CN",
      "tags": ["killsay", "community"],
      "serverTags": ["generic"],
      "version": 1,
      "updatedAt": "2026-05-26T00:00:00Z",
      "entryCount": 3,
      "sha256": "这里放对应文本包文件的 sha256",
      "downloadUrl": "https://raw.githubusercontent.com/bedkillerspacex-boop/KsText/master/packs/%E6%83%85%E8%AF%9D%E7%89%88.json"
    }
  ]
}
```

### 每个字段是什么意思

- `id`
  - 包的唯一 ID
- `name`
  - 显示名
- `author`
  - 作者名，没有就写 `佚名`
- `summary`
  - 简单介绍，几十个字以内就够
- `language`
  - 中文包就写 `zh-CN`
- `tags`
  - 搜索标签，一般保留 `killsay` 和 `community`
- `serverTags`
  - 服务器标签，没有特别分类就写 `generic`
- `version`
  - 这个包自己的版本号
  - 改过一次内容就可以加 1
- `updatedAt`
  - 这条记录最后更新时间
- `entryCount`
  - 这个包里一共有多少条文案
- `sha256`
  - 对应 `packs/*.json` 文件内容的 sha256
- `downloadUrl`
  - 这个包的原始下载地址
  - 必须能直接打开到 JSON 文件

## 5. 最常见的维护任务

## 5.1 新增一个文本包

按这个顺序做：

1. 在 `packs/` 里新建一个中文文件名的 `.json`
2. 按正确格式写入 `id`、`name`、`entries`
3. 数一遍 `entries` 里有多少条
4. 计算这个文件的 `sha256`
5. 打开 `index.json`
6. 在 `packs` 数组里新增一条对应记录
7. 把 `downloadUrl` 写成这个新文件的 raw 地址
8. 检查分支名是不是 `master`
9. 保存后提交推送

## 5.2 修改一个已有文本包

按这个顺序做：

1. 直接改对应的 `packs/*.json`
2. 改完后重新数 `entries`
3. 重新计算 `sha256`
4. 到 `index.json` 里同步更新：
   - `entryCount`
   - `sha256`
   - `updatedAt`
   - `version`
5. 如果名字、简介、作者也变了，就一起改
6. 保存后提交推送

## 5.3 删除一个文本包

按这个顺序做：

1. 删除 `packs/` 里的对应文件
2. 打开 `index.json`
3. 把这条包记录整个删掉
4. 保存后提交推送

## 5.4 重命名一个文本包

不要只改文件名，必须全套一起改。

按这个顺序做：

1. 重命名 `packs/旧名字.json` 到 `packs/新名字.json`
2. 打开这个包文件
3. 视情况同步改里面的 `id` 和 `name`
4. 打开 `index.json`
5. 改这条记录里的：
   - `id`
   - `name`
   - `downloadUrl`
   - `sha256`
   - `updatedAt`
   - `version`
6. 保存后提交推送

## 6. 怎么算 `entryCount`

很简单：

- `entries` 数组里有几条，就写几

例如：

```json
"entries": [
  "第一条",
  "第二条",
  "第三条"
]
```

那 `entryCount` 就是：

```json
"entryCount": 3
```

不要瞎填。
列表显示条数不对，大概率就是这里没改。

## 7. 怎么算 `sha256`

在 PowerShell 里可以这样算：

```powershell
Get-FileHash .\packs\情话版.json -Algorithm SHA256
```

你会得到一串十六进制结果，把它填进 `index.json` 的 `sha256` 字段。

注意：

- 文件内容只要改了一个字，`sha256` 就会变
- 所以改过文本包后，`sha256` 必须重算

## 8. `downloadUrl` 怎么写

格式照着这个写：

```text
https://raw.githubusercontent.com/bedkillerspacex-boop/KsText/master/packs/文件名.json
```

如果文件名里有中文，最终 URL 里会变成编码后的样子，这是正常的。

例如：

```text
https://raw.githubusercontent.com/bedkillerspacex-boop/KsText/master/packs/%E6%83%85%E8%AF%9D%E7%89%88.json
```

记住两个重点：

- 路径要指向真实存在的文件
- 分支要和仓库实际使用的分支一致，目前是 `master`

## 9. 提交前检查清单

每次推送前，把下面这几条过一遍：

- 新增的包文件是不是放在 `packs/` 里
- 文件名是不是中文且能看懂
- 包文件是不是合法 JSON
- `entries` 有没有空内容
- `index.json` 里有没有对应记录
- `entryCount` 对不对
- `sha256` 是不是最新的
- `downloadUrl` 能不能对应到真实文件
- `downloadUrl` 的分支是不是 `master`
- 作者没写的时候是不是填了 `佚名`

## 10. 提交和推送命令

在仓库根目录执行：

```powershell
git status
git add .
git commit -m "update packs"
git push origin master
```

如果你这次只是补文档，也可以写：

```powershell
git commit -m "add maintenance guide"
```

## 11. 最容易出错的地方

下面这些是最容易把仓库搞坏的点：

### 只改了 `packs/*.json`，没改 `index.json`

后果：

- mod 搜不到新包
- 或者条数显示不对
- 或者下载到旧信息

### `downloadUrl` 写错分支

后果：

- 列表能同步
- 但是点包加载直接失败

### `entryCount` 没更新

后果：

- 列表显示条数不对

### `sha256` 没更新

后果：

- 后续如果要做校验，会直接对不上

### 文件不是合法 JSON

后果：

- mod 读不到
- GitHub raw 能开，程序也可能解析失败

## 12. 最稳的维护习惯

最稳的做法就是每次只做一件事：

- 要么新增一个包
- 要么修改一个包
- 要么删除一个包

不要一次性乱改几十个名字、几十个链接、几十个介绍。

改完就检查：

1. `packs/` 对不对
2. `index.json` 对不对
3. `downloadUrl` 对不对
4. 再提交

## 13. 一句话总结

维护这个仓库其实就三件事：

1. 文本包文件放进 `packs/`
2. `index.json` 把它登记好
3. `downloadUrl` 和分支别写错

只要这三件事没错，mod 基本就能正常用。
